# systems

Low-level and systems-programming projects — operating systems, kernels, concurrency, and advanced data structures. Mostly C.

Each linked doc below was written by **reading the actual source**, not the README — including computed line counts and an explicit "README vs. code" section that flags where the marketing and the implementation diverge.

| Project | What it actually is | Language / size | Notable code-level finding |
|---------|---------------------|-----------------|----------------------------|
| [VeridianOS](VeridianOS.md) | Bare-metal x86-64 **type-1 hypervisor microkernel** — every OS component runs as a hardware-isolated VM-guest domain (VMX/SVM, EPT/NPT, COW transactions, Linux syscall ABI, from-scratch TCP/IP, IOMMU) | C + asm · **70,788 LOC / 203 src files** | Real and deep; README is unusually honest. Minor undercounts (claims ~50 syscalls, 70 defined; 199 files vs 203); caps are *not* time-limited despite the claim; no in-tree Linux-guest boot path; branch is `master`. |
| [TaskForge-OS](TaskForge-OS.md) | An **in-process OS simulation** in C that runs a banking app through inlined "system calls" (schedulers, paging, deadlock logic) | C · ~11.7k C LOC | Not a bootable kernel — "syscalls" are `static inline` C calls. `kernel_schedule_next` is implemented but **never called** by the banking path; boot banner prints "Banker's prevention: ON" while the C flag stays `0`. |
| [SyncUp](SyncUp.md) | A concurrent **one-way file-backup engine** in C (pthread worker pool, slab allocator, resource-allocation-graph deadlock tracking) wrapped in an OS-course demo suite | C · 5,453 LOC (+1,992 Py GUI) | "Multiprocessing via `fork()`" is `#ifdef`-stubbed to single-process on Windows (all children share one PID); the queue is a max-heap not the claimed min-heap; CRC-32 checksum is dead code; it's a mirror, not a true bidirectional sync. |
| [GridWatch](GridWatch.md) | An **emergency-dispatch simulator** where every on-screen metric is powered by a different advanced data structure; clean C ABI + Win32 TUI + Flask/ctypes web UI | C · 8,201 LOC / 49 files | Fibonacci heap, suffix array (Kasai), BK-tree, segment tree, succinct bitvector etc. are genuinely implemented **and queried by the running sim**; ~14 *other* structures are built and kept in sync but their query side is only exercised by unit tests. |
| [OS-Exam-Prep](OS-Exam-Prep.md) | A dependency-free **static study SPA** (one `index.html` + a generated `content.js`) compiled by stdlib-only Python from Markdown notes | HTML/JS + Python · 12,922 content LOC | 36 Markdown files → 62 model answers + 20 **static** (not interactive) mind maps. The generated `content.js` is uncommitted and not gitignored, so a fresh clone is broken until `build_site.py` runs. |
| [Smart-Dictionary](Smart-Dictionary.md) | A **BST vs AVL vs threaded-BT** benchmark behind a word autocomplete engine | C · 3,175 LOC | All three trees real (BST uses Morris traversal + DSW balancing); **no trie**. The benchmark measures synthetic 500–5k shuffled words, *not* the 90k corpus, and never triggers BST's worst case; `docs/report.md` is empty so no results are stored; an 884-line GTK3 GUI goes undocumented. |
